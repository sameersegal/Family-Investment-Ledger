# Bank Statement Parser to Neo Ledger Mapping

This document defines the transformation contract from the sibling project `bank-statement-parser` into Neo Ledger input tables.

It covers:

- `position_transactions` -> `Trades`
- `cash_transactions` -> `CashMovements`
- synthetic trade cash legs -> `CashMovements`

## Goal

The parser output is semantically close to Neo Ledger, but not schema-compatible as-is.

The adapter layer must:

1. enrich parser records with source-specific static metadata
2. map parser fields to Neo Ledger fields
3. generate stable IDs
4. derive missing fields such as `FXRateToINR`
5. emit synthetic `CashMovements` for trade cash effects so `computeCashBalances()` remains correct

## Source Contract

Expected parser JSON shape:

```json
{
  "broker": "ibkr | schwab",
  "account_id": "string",
  "period": {
    "from": "YYYY-MM-DD",
    "to": "YYYY-MM-DD"
  },
  "position_transactions": [
    {
      "date": "YYYY-MM-DD",
      "type": "buy | sell",
      "ticker": "AAPL",
      "quantity": 10,
      "price": 200,
      "amount": -2000,
      "currency": "USD"
    }
  ],
  "cash_transactions": [
    {
      "date": "YYYY-MM-DD",
      "type": "dividend | interest | fee | deposit | withdrawal | forex | tax | other",
      "description": "string",
      "amount": 100,
      "currency": "USD"
    }
  ]
}
```

## Neo Ledger Target Tables

### Trades

Canonical fields used by Neo Ledger:

| Field | Required | Notes |
|---|---|---|
| `TradeId` | yes | Generated by adapter |
| `TradeDate` | yes | Trade date |
| `OwnerId` | yes | Usually static per source/account |
| `BrokerId` | yes | Usually static per source/account |
| `AccountId` | yes | Usually static per source/account |
| `SecurityId` | yes | Must match `Securities.SecurityId` |
| `Side` | yes | `BUY` or `SELL` |
| `Quantity` | yes | Positive number |
| `Price` | yes | Native currency price per unit |
| `Fees` | yes | Use `0` if unavailable |
| `FXRateToINR` | yes | `1` for INR assets, otherwise enriched |
| `Notes` | optional | Free text |
| `SourceRef` | optional | Statement or parser provenance |

### CashMovements

Canonical fields used by Neo Ledger:

| Field | Required | Notes |
|---|---|---|
| `CashTxnId` | yes | Generated by adapter |
| `TxnDate` | yes | Cash transaction date |
| `OwnerId` | yes | Usually static per source/account |
| `AccountId` | yes | Usually static per source/account |
| `Currency` | yes | Native currency |
| `Amount` | yes | Signed amount |
| `Category` | yes | Adapter-mapped enum |
| `LinkedTradeId` | optional | Used for synthetic trade cash legs |
| `LinkedActionId` | optional | Empty for parser flow |
| `IsForeignIncome` | yes | Derived rule |
| `Notes` | optional | Description or generated note |
| `SourceRef` | optional | Statement or parser provenance |

## Adapter Configuration

Some metadata is known a priori for a given source and should be configured, not inferred from each row.

Recommended source config shape:

```json
{
  "sourceId": "ibkr_u1234567",
  "parserBroker": "ibkr",
  "parserAccountId": "U1234567",
  "ownerId": "SS",
  "brokerId": "IBKR",
  "accountId": "IBKR_MAIN",
  "defaultTradeFee": 0,
  "defaultCashIsForeignIncome": false,
  "securityMap": {
    "AAPL": "AAPL",
    "GOOG": "GOOG",
    "912797TB3": "912797TB3"
  },
  "currencyToINR": {
    "INR": 1
  }
}
```

### Required Static Inputs

| Config Field | Purpose |
|---|---|
| `ownerId` | Fills Neo Ledger `OwnerId` |
| `brokerId` | Fills Neo Ledger `BrokerId` |
| `accountId` | Fills Neo Ledger `AccountId` |
| `securityMap` | Converts parser ticker to Neo Ledger `SecurityId` |
| `defaultTradeFee` | Used when parser does not emit fees |
| `currencyToINR` or FX lookup | Provides `FXRateToINR` |

## Mapping Rules

### 1. Parser `position_transactions` -> Neo Ledger `Trades`

| Parser Field | Neo Field | Rule |
|---|---|---|
| `date` | `TradeDate` | Copy as-is |
| `type` | `Side` | `buy` -> `BUY`, `sell` -> `SELL` |
| `ticker` | `SecurityId` | Map via `securityMap`; fail if missing |
| `quantity` | `Quantity` | Copy as positive number |
| `price` | `Price` | Copy as native price |
| `currency` | none directly | Used to derive `FXRateToINR` |
| top-level `broker` | `BrokerId` | Prefer static config mapping |
| top-level `account_id` | `AccountId` | Prefer static config mapping |
| static config | `OwnerId` | Fill from config |
| static config or rule | `Fees` | Default to `0` if unavailable |
| FX lookup | `FXRateToINR` | Required enrichment |
| generated | `TradeId` | Stable generated ID |
| generated | `Notes` | Optional provenance note |
| generated | `SourceRef` | Statement-level reference |

#### Trade Rules

1. `quantity` must remain positive for both buys and sells.
2. `amount` from parser is not stored in `Trades`, but should be validated against `quantity * price`.
3. `amount < 0` is expected for `buy`; `amount > 0` is expected for `sell`.
4. `ticker` must resolve to an existing Neo Ledger security.
5. `FXRateToINR` is mandatory for all trades used in tax/gain calculations.

#### Trade Validation

For each parser trade:

```text
expectedAmount = quantity * price
```

Accept if:

- `abs(abs(amount) - expectedAmount) <= tolerance`

Recommended tolerance:

- `0.01` for normal equities
- configurable for bond lots if statements round unusually

#### Trade ID Format

Recommended stable format:

```text
TRD_{sourceId}_{YYYYMMDD}_{ticker}_{side}_{sequence}
```

Example:

```text
TRD_ibkr_u1234567_20250915_GOOG_BUY_001
```

### 2. Parser `cash_transactions` -> Neo Ledger `CashMovements`

| Parser Field | Neo Field | Rule |
|---|---|---|
| `date` | `TxnDate` | Copy as-is |
| `amount` | `Amount` | Copy signed value |
| `currency` | `Currency` | Copy as-is |
| `description` | `Notes` | Copy as-is |
| `type` | `Category` | Map through category table |
| static config | `OwnerId` | Fill from config |
| static config | `AccountId` | Fill from config |
| generated | `CashTxnId` | Stable generated ID |
| generated | `LinkedTradeId` | Empty |
| generated | `LinkedActionId` | Empty |
| derived | `IsForeignIncome` | Based on category and account context |
| generated | `SourceRef` | Statement-level reference |

#### Cash Category Mapping

| Parser `type` | Neo `Category` | Default `IsForeignIncome` |
|---|---|---|
| `dividend` | `DIVIDEND` | `TRUE` for foreign accounts, else `FALSE` |
| `interest` | `INTEREST` | `TRUE` for foreign accounts, else `FALSE` |
| `tax` | `TAX` | `FALSE` |
| `fee` | `FEE` | `FALSE` |
| `deposit` | `DEPOSIT` | `FALSE` |
| `withdrawal` | `WITHDRAWAL` | `FALSE` |
| `forex` | `FOREX` | `FALSE` |
| `other` | `OTHER` | `FALSE` unless overridden |

#### Cash Rules

1. Keep parser sign convention unchanged.
2. `IsForeignIncome` should only be true for income-like inflows that matter to RBI ageing.
3. Taxes and fees should not be marked as foreign income even on foreign accounts.
4. If the parser later emits richer tax subtypes, adapter rules can refine `Category` without changing Neo Ledger schema.

#### CashTxnId Format

Recommended stable format:

```text
CASH_{sourceId}_{YYYYMMDD}_{category}_{sequence}
```

Example:

```text
CASH_ibkr_u1234567_20250915_DIVIDEND_001
```

### 3. Synthetic Trade Cash Legs -> Neo Ledger `CashMovements`

This step is required if Neo Ledger cash balances continue to be computed only from `CashMovements`.

Why:

- parser trade cash effects live in `position_transactions`
- Neo Ledger cash balances only sum `CashMovements`
- therefore buys and sells must also create cash rows

#### Synthetic Mapping

| Source | Neo Field | Rule |
|---|---|---|
| trade `date` | `TxnDate` | Same as `TradeDate` |
| trade `currency` | `Currency` | Same as trade currency |
| trade `amount` | `Amount` | Copy parser signed amount |
| generated | `Category` | `BUY_SETTLEMENT` or `SELL_PROCEEDS` |
| generated | `LinkedTradeId` | Link to generated `TradeId` |
| generated | `LinkedActionId` | Empty |
| static config | `OwnerId` | Fill from config |
| static config | `AccountId` | Fill from config |
| generated | `Notes` | e.g. `Synthetic cash leg for BUY AAPL` |
| generated | `IsForeignIncome` | `FALSE` |
| generated | `SourceRef` | Same statement provenance |

#### Synthetic Category Rules

| Trade Side | Cash Category | Expected Amount Sign |
|---|---|---|
| `BUY` | `BUY_SETTLEMENT` | negative |
| `SELL` | `SELL_PROCEEDS` | positive |

#### Fees Handling

If parser fees are unavailable:

1. keep `Trades.Fees = 0`
2. do not generate separate fee cash rows

If parser fees become available later:

1. set `Trades.Fees`
2. optionally emit a separate `CashMovements` row with `Category = FEE`
3. avoid double-counting if trade `amount` already includes fees

  "ticker": "GOOG",

## Validation Checklist

### Record-Level Validation

1. every parser `position_transaction` maps to exactly one Neo Ledger `Trades` row
2. every parser `position_transaction` also maps to exactly one synthetic `CashMovements` row
3. every parser `cash_transaction` maps to exactly one Neo Ledger `CashMovements` row
4. every mapped `SecurityId` exists in `Securities`
5. every emitted `AccountId` exists in `Entities` with `EntityType = ACCOUNT`
6. every emitted `OwnerId` and `BrokerId` matches that account configuration
7. every non-INR trade has a non-null `FXRateToINR`

### Statement-Level Validation

For each statement:

```text
sum(synthetic trade cash legs) + sum(mapped cash transactions)
```

should equal:

```text
sum(parser position_transactions.amount) + sum(parser cash_transactions.amount)
```

This confirms the adapter preserves parser economics.

### Neo Ledger Compatibility Validation

After import:

1. `rebuildLots()` should succeed with no missing `SecurityId`
2. `computeCashBalances()` should include both non-trade cash and trade settlement effects
3. `computeRBI180DayExposure()` should only age qualifying foreign income inflows

## Known Open Questions

These should be confirmed before implementation:

1. exact allowed `CashMovements.Category` vocabulary in production data
2. preferred `SourceRef` convention
3. authoritative FX source for `FXRateToINR`
4. whether trade `amount` from each broker is gross or net of fees
5. whether `SecurityId` should always equal ticker or may require exchange-prefixed ids

## Recommended First Implementation Scope

To keep the first adapter narrow and reviewable:

1. support one source configuration at a time
2. require explicit `securityMap`
3. require explicit `ownerId`, `brokerId`, and `accountId`
4. use `Fees = 0` unless manually supplied
5. use a provided FX lookup table for trade dates
6. emit both `Trades` and `CashMovements`

This gives a correct ingestion path without changing Neo Ledger core logic.
